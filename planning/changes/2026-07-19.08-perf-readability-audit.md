# Design: Perf & readability refactor audit (both-axes, decision-grade)

## Summary

Run a two-lens multi-agent audit of the `modern_di/` source and produce one
ranked findings report at `planning/audits/2026-07-19-perf-readability-audit-report.md`.
It reuses the shape of the committed `bug-hunt-audit.workflow.mjs` harness
(Discover → parallel Find → adversarial Verify → Synthesize-and-write), retargeted
to two lenses — **performance** (bench-mapped hypotheses) and
**readability/structure** — and gated against the already-settled corpus
(`planning/decisions/` + `deferred.md`) so nothing already tried-and-dropped
resurfaces without new evidence. No code changes ship from this audit; the
report is the sole deliverable, plus the reusable workflow script.

## Motivation

The resolve hot path is at a documented floor: the single-path compiled resolver
shipped (#334), the warm-singleton memo-swap was built/measured (~1.6x) and
dropped (`deferred.md`), child lazy-alloc was declined on measurement (#355), the
`exec`-codegen ceiling is a deliberate zero-dep stance, and free-threaded
non-scaling is diagnosed as CPython refcount contention, not liftable here. So
the value of a "make it faster / cleaner" pass is **not** finding easy wins — it
is a decision-grade survey that (a) separates genuinely-new perf hypotheses from
settled ground, and (b) finds off-hot-path readability seams that don't regress
the numbers already fought for. On the hot path readability and speed are in
deliberate tension (the inlined override-guard/navigate/closed-check preamble is
copy-pasted across all six compiled closures on purpose to hold the per-node
frame at 1); the open readability ground is off that path.

## Design

Execution vehicle: a committed Workflow script
`planning/scripts/perf-readability-audit.workflow.mjs`, adapted from the
bug-hunt harness. Four phases:

**Discover.** Build the file map (path, line count, one-line role) for
`modern_di/**`, plus the **settled corpus**: each `planning/decisions/*` ruling
(slug + one-line holding), each `deferred.md` item with its revisit-trigger, the
guard-scenario catalog (G1-G15 from `benchmarks/README.md`), and the current
competitive table from `deferred.md`. This corpus is the guardrail every finder
and verifier cites. No analysis in this phase.

**Find — two lenses (parallel):**

- *Performance.* Each finding is a **hypothesis**, not a claim: it names the
  guard scenario (G1-G15) that would confirm it, an expected-leverage estimate,
  and the specific `deferred.md`/decision it must clear. No prototyping, no
  bench run. The prompt states the no-`exec` / zero-dep / conservative-feature
  stances as settled — a finding may note them but must not re-propose them as if
  open.
- *Readability/structure.* A concrete `file:line` seam or simplification,
  **weighted to complexity**: full-depth on `exceptions.py`, `container.py`,
  `resolver_compiler.py`, `factory.py`, `dependency_graph.py`, `wiring.py`,
  `types_parser.py`; a lighter confirmation pass on the small stable files. Each
  finding is tagged on/off hot path and names the invariant it must not break
  (per-node frame count, 100% coverage, zero-dep, behavior).

**Verify — three lenses per finding, majority vote (≥2 confirms to survive):**

1. *read-real-code* — the cited code actually does what the finding claims
   (catches hallucinated file:line / evidence).
2. *decision-conflict* — is this already ruled on in `decisions/`/`deferred.md`?
   If yes, refute unless the finding carries genuinely new evidence; cite the
   settled item. This lens is the mature-repo adaptation that keeps the report
   from re-litigating.
3. *invariant-safety / leverage-realism* — readability: does the change provably
   preserve behavior + frame count + coverage? performance: is the guard-scenario
   mapping honest and the leverage estimate defensible rather than hand-waved?

**Synthesize.** Dedup across lenses, then bucket survivors on a
**leverage-vs-risk grid** and write the report:

- `do-first` — high leverage, low risk.
- `needs-decision` — high leverage, high risk (a maintainer call: an invariant
  or stance is in play).
- `cleanup` — low leverage, low risk.
- `skip` — low leverage, high risk.
- `already-settled` — matched an existing decision/deferred item; recorded with
  the citation so the next audit doesn't re-raise it.

Perf hypotheses and readability-actionables live in separate report sections
because their evidence bars differ (hypothesis+guard-map vs concrete file:line).
The report header carries the baseline commit and links this spec.

**Model routing** (smaller models where possible): the cheap model for Discover,
both finders, and the routine verify lenses; the strong model only for
Synthesize and any finding whose three verifier votes split.

## Non-goals

- No code changes ship from this audit — report only.
- Sibling repos (modern-di-pytest, framework integrations) out of scope.
- `exec`/codegen and the other settled stances are noted-as-settled, never
  proposed as open.
- Lint/style findings excluded — ruff (`select = ["ALL"]`) and ty own those.
- No prototyping or bench runs; perf findings stay hypotheses (per scope ruling).

## Testing

Not a code change, so no test gate. "Landed correctly" means: the report exists
at the target path, every finding cites a real `file:line` (spot-check 3-5), the
`already-settled` bucket correctly quotes the decision/deferred item it matches,
and every perf finding names a real G-scenario. `just check-planning` passes with
this change file present.

## Risk

- **Re-surfacing settled ground** (likelihood: high without the gate, low with
  it). Mitigation: the Discover settled-corpus + the decision-conflict verify
  lens; `already-settled` is a first-class bucket, not a silent drop.
- **Perf hypotheses read as commitments.** Mitigation: they are explicitly
  labeled hypotheses with a named confirming scenario and are never actioned by
  this audit.
- **Finder hallucination** (a known multi-agent failure mode). Mitigation: the
  read-real-code lens + majority vote, same discipline as the bug-hunt harness.
- **Cost.** Two lenses (not four) and cheap-model fan-out keep it well under the
  bug-hunt run.
