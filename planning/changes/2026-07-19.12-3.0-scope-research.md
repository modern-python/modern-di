# Design: 3.0-scope research pass (UX-weighted)

## Summary

Run a UX-weighted multi-agent research pass over the modern-di core (and the
integration surface it exposes) and produce one report at
`planning/audits/2026-07-19-3.0-scope-research-report.md`. It reuses the
`perf-readability-audit.workflow.mjs` shape, retargeted to three lenses —
UX/API/DX (primary), integration UX, and a light perf+readability
re-confirmation — gated against the prior-candidate ledger from the
[2026-07-05 v3-ux report](../audits/2026-07-05-v3-ux-research-report.md) plus
`decisions/`, `deferred.md`, and the five queued 3.0 switches. Findings land on
a leverage-vs-risk grid with a 3.0-relevance cross-tab (breaking / non-breaking /
post-3.0). No code changes; the report feeds a resumed 3.0-scope brainstorm.

## Motivation

3.0's breaking core is already the five queued switches (`migration/to-3.x.md`),
each warned in 2.x. Before committing what else rides the one-time
breaking-change budget, the maintainer wants fresh evidence across UX,
performance, and readability. Perf is at its documented floor and readability was
just audited and actioned this session (PRs #356-358), so those two are light
re-confirmation only. The fresh, high-leverage axis for a major is UX/API/DX,
where the existing v3-ux report is two weeks stale and much has shipped since.

## Design

Execution vehicle: a committed Workflow script
`planning/scripts/3.0-scope-research.workflow.mjs`, adapted from the
perf-readability harness. Four phases:

**Discover.** File map of `modern_di/**` and the docs/integration surface, plus
the **prior-candidate ledger**: all 30 candidates from the v3-ux report with
id/title/ruling (shipped / accepted / declined / deferred), cross-checked against
current code to mark what already shipped since 2026-07-05; the five queued
switches; `decisions/` holdings; and `deferred.md` items. This ledger is the
guardrail that stops the pass re-surfacing settled or shipped conclusions.

**Find — three lenses:**

- *UX / API / DX* (primary, full-depth) — API ergonomics, error/diagnostic
  quality, docs/onboarding, surprising defaults. Each finding carries a
  3.0-relevance tag (breaking / non-breaking / post-3.0 / n-a).
- *Integration UX* — the `@inject` asymmetry (7 of 12 integrations require it),
  blessed-readiness (A2), one-import setup. Worked from the v3-ux §3.4 +
  `architecture/integration-kit.md` + `modern_di/integrations.py`. The sibling
  integration repos are **not in this working tree**, so these findings are
  flagged lower-confidence / "needs sibling-repo confirmation."
- *Perf + readability re-confirm* (light) — leans on this session's merged audit;
  reports only something new, regressed, or newly-enabled by a breaking change,
  not a redo.

**Verify — three lenses, majority vote (≥2 to survive):**

1. *read-real-code* — the cited surface exists and behaves as claimed.
2. *prior-art-conflict* — already shipped or ruled in the ledger / decisions /
   deferred? If so, refute unless the finding brings new evidence; cite the item.
3. *ux-realism / leverage* — real friction backed by evidence (a signature, an
   error string, a doc gap), not opinion; leverage estimate defensible.

**Synthesize.** Dedup, then bucket survivors on the leverage-vs-risk grid
(do-first / needs-decision / cleanup / skip / already-settled) and additionally
cross-tab by 3.0-relevance so the report reads straight into the scope decision.
Integration findings kept in their own section with the confidence caveat.
Perf/readability re-confirm findings segregated.

**Model routing** (smaller where possible): the cheap model for Discover, the
three finders, and the routine verify lenses; the strong model only for
Synthesize.

## Non-goals

- No code changes and no scope commitment — the report informs a resumed
  brainstorm; it does not decide 3.0.
- Not re-auditing perf (at floor) or readability (actioned this session) in full.
- Not reading sibling integration repo source (not in tree); integration
  findings lean on prior research + core seams and say so.
- The five queued switches are settled facts, not findings.

## Testing

Not a code change. "Landed correctly" means: the report exists at the target
path; every finding cites a real surface (spot-check 3-5); the already-settled
bucket correctly cites the ledger/decision it matches; each UX finding carries a
3.0-relevance tag; integration findings carry the confidence caveat.
`just check-planning` passes with this change file present.

## Risk

- **Re-surfacing the v3-ux report's own conclusions** (high without the gate).
  Mitigation: the Discover ledger + the prior-art-conflict verify lens;
  already-settled is a first-class bucket.
- **Integration findings overreach** (can't read the repos). Mitigation: explicit
  lower-confidence flag and a dedicated report section.
- **Opinion masquerading as UX finding.** Mitigation: the ux-realism lens
  requires a concrete cited surface, not a preference.
- **Cost.** Three lenses with cheap-model fan-out; the heavy Discover ledger is
  one agent.
