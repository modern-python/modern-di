# Design: modern-di 3.0 release scope

## Summary

3.0 flips the five queued switches (each already warned in 2.x) from
warn-then-continue to raise/validate. The one design change is **validate-by-default,
deferred to `open()`/first-resolve** rather than construction, so an
integration's connection `ContextProvider`s (registered *after* the user builds
the container) are in the graph before validation runs. `validate=` collapses to
on/off: `True` and the default (`None`) both defer; `False` is off;
construction-time validation stays available via an explicit `container.validate()`
call. The research pass ([2026-07-19-3.0-scope-research-report](../audits/2026-07-19-3.0-scope-research-report.md))
confirmed **nothing else earns the breaking budget** (0 breaking findings). Two
non-breaking docs cleanups it surfaced are out of scope for the major.

## Motivation

The 3.0 breaking core has been signalled in 2.x for a full cycle
([migration/to-3.x.md](../../docs/migration/to-3.x.md)); the migration guide is
written. The only unresolved design question was switch #4's integration
footgun: with `validate()` at construction, a `Factory` depending *by type* on
`fastapi.Request`/`taskiq.TaskiqMessage`/… raises `ArgumentResolutionError`
because the integration wires that `ContextProvider` *after* construction. The
maintainer ruled Option B (defer validation to when the graph is assembled),
both-deferred semantics.

## Design

**The five switches** (implement the already-documented behavior; the error
classes `ContainerClosedError` / `ContextValueNotSetError` already exist):

1. Reusing a closed container raises `ContainerClosedError` (drop the self-heal).
2. `Alias(scope=)` parameter removed.
3. `Factory(cache_settings=)` removed.
4. `validate()` on by default, **deferred** (below).
5. Direct resolve of an unset `ContextProvider` raises `ContextValueNotSetError`.

**Validate deferral (switch #4).** Root cause: the graph is incomplete at
`__init__` because integrations register context providers afterward. So:

- `validate=None` (default) and `validate=True` → validation enabled, **deferred**;
  `validate=False` → off. `True` and the default are identical; `validate=` is
  on/off.
- Trigger: `open()` runs `self.validate()` once when enabled and not yet
  validated (root-only — `parent_container is None`; children never validate,
  unchanged). A first-resolve fallback in `resolve_provider` covers a container
  never entered via `with`/`open()` — a single guarded bool check at the
  top-level entry (not per-node), cheaper than the `_warn_and_reopen_if_closed`
  call already there.
- Once-only: guarded by `_validated`, which persists across `close()` (close only
  sets `closed=True`). So **a plain close→reopen never re-walks the graph**; the
  only re-trigger is a genuine `add_providers` mutation (root-only,
  already re-validates today).
- Remove the eager `if validate: self.validate()` from `__init__` and drop the
  `UnvalidatedContainerWarning` (3.0 validates by default; nothing to warn).

**Docs.** Revise `migration/to-3.x.md` switch #4: "raises at construction time"
→ "at container entry (`open()`/`with`) or first resolve," and note that
`validate=True` is no longer eager (call `container.validate()` for
construction-time failure). Update the 2.x `UnvalidatedContainerWarning` /
`exceptions.py` wording that promises "at root construction" accordingly. Promote
the `validate()` timing into `architecture/containers.md` and
`architecture/validation.md`.

**Sequencing.** Implement as per-switch change bundles + one validate-deferral
bundle, converging on `main`; then cut a `3.0.0rc1` pre-release (PEP 440),
smoke-test the sibling integrations against it, then tag `3.0.0`. Integrations
need **no code changes** — they rely on the default, which now completes the
graph before validating (the Option B win); only release-notes timing guidance.

## Non-goals

- No new `validate` tri-state/enum value — it stays `bool | None`, on/off.
- The two non-breaking docs cleanups (positional-only error wording;
  `ContextProvider scope=UNSET` docs) ship independently, not in the major.
- No eager cache warm-up, no `exec`/codegen, no async resolution, no new provider
  types — settled stances the research re-confirmed.
- No behavior change to switches 1-3 and 5 beyond removing the deprecated
  path/param.
- No sibling-integration code changes; this spec is core + docs only.

## Testing

- Per switch: the 3.0 behavior raises — closed-container → `ContainerClosedError`;
  unset-context direct resolve → `ContextValueNotSetError`; `Alias(scope=)` /
  `Factory(cache_settings=)` → `TypeError` (unknown kwarg).
- Validate deferral: (a) an invalid graph does **not** raise at construction but
  raises at `open()`/first resolve; (b) **reopen does not re-walk** (lock the
  once-only guard — construct, enter, close, re-enter: `validate()` runs once);
  (c) `validate=False` never validates; (d) the integration pattern — register a
  `ContextProvider` post-construction, then `open()` — no false
  `ArgumentResolutionError`; (e) `add_providers` after first validation
  re-validates.
- `just test-ci` green at 100% coverage; the migration guide's
  warnings-as-errors readiness recipe still yields a clean upgrade.

## Risk

- **`validate=True` timing change** (2.x eager → 3.0 deferred): a documented
  major change; mitigation = call `container.validate()` for construction-time.
- **Never-entered container skips validation**: the first-resolve fallback covers
  the realistic path; a constructed-but-never-resolved container is degenerate.
- **Per-resolve guard on the perf-sensitive path**: one predictable bool at
  `resolve_provider` entry (not per-node); negligible beside the existing
  entry-time closed check — but noted, given the hot path's sensitivity.
- **Cross-repo**: integrations depend on the default; the `rc` smoke test
  verifies no integration passed `validate=True` in a way that reintroduces the
  footgun before the tag.
