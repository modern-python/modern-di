# Design: Flat change files

## Summary

The planning convention (canonical repo `lesnik512/planning-convention`) gains
a 2.0.0 release that removes the folder around change specs: a change is a
single file `changes/YYYY-MM-DD.NN-<slug>.md`, written from either the design
or the change template. modern-di then applies the update, `git mv`-ing each
bundle's spec file up one level. Major version because the structure is
breaking for consumers: the validator rejects the old folder form.

## Motivation

The folder earned its keep when a bundle was a set (`design.md` + `plan.md`).
Since 1.2.0 (ephemeral plans) every bundle holds exactly one file — a
directory wrapping one file is the same ceremony 1.2.0 removed. Evidence the
extension point was never used: `ALLOWED_BUNDLE_FILES` only ever named the
three spec files, and none of modern-di's 41 bundles ever carried another
file. Flat files also make `changes/` symmetric with `decisions/`.

## Design

- **Shape**: `changes/YYYY-MM-DD.NN-<slug>.md`. The `.NN` intra-day counter
  and slug rules are unchanged. The lane (Full vs Lightweight) remains a
  process choice of which template to start from; it is no longer visible in
  any filename, and nothing consumes it — `index.py` reads only `summary`.
- **Attachments**: none, by design. Non-prose assets are inlined (tables,
  mermaid) or belong in `audits/`. A directory under `changes/` is a
  validation error with a message pointing at the 2.0.0 migration.
- **`index.py`**: `load_bundles` globs `changes/*.md` (skipping `README.md`
  and `_`-prefixed files, mirroring `load_decisions`) and drops the
  design-over-change preference; `ALLOWED_BUNDLE_FILES` and the
  unexpected-file check are deleted; `check` validates the filename stem
  against the existing `YYYY-MM-DD.NN-slug` pattern, requires `summary`, and
  flags any directory or non-`.md` file under `changes/`.
- **Prose**: convention.md's Quick path step 2 collapses to "create the file
  from the matching template"; "bundle" wording becomes "change file"
  throughout; the Frontmatter section merges the design.md/change.md line
  into one change-file line. Templates keep their content; only their
  role line changes (they are now the file, not a file inside a folder).
- **`APPLY.md` / `CHANGELOG`**: 2.0.0 entry carries the consumer migration —
  one `git mv "$d/design.md" "${d%/}.md"` (or `change.md`) per bundle dir,
  then remove the empty dirs; `just check-planning` finds stragglers.
- **modern-di apply**: after PR #278 merges — vendor the new `index.py`,
  merge the new prose into `planning/README.md`, flatten all 41 bundles with
  `git mv` (history survives via `--follow`), commit this spec as a flat
  file, bump `.convention-version` to `2.0.0`.

## Non-goals

- No change to `decisions/`, `audits/`, `releases/`, `retros/`, templates'
  content, or the two-axis model.
- No dual-form support: the validator accepts only flat files. Rejected
  alternative (accept both forms to avoid migration) traded a five-minute
  scripted migration for permanent two-shape complexity.

## Testing

- Canonical: pytest suite updated — flat files validate; a directory under
  `changes/` yields the migration-pointing violation; name/`summary` checks
  ported; 100% coverage gate holds.
- modern-di: `just check-planning` prints `planning: OK` after the flatten;
  `just index` renders all 41 rows with correct dates/slugs; `just lint-ci`
  passes.

## Risk

- Consumers on 1.x fail `--check` after updating until they run the
  migration — intended; the CHANGELOG entry is the migration note and the
  violation message names it.
- Historical `git log` on a flattened file needs `--follow` — accepted;
  noted in the CHANGELOG entry.
