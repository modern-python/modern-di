# Design: Rebuild the benchmark suite around a hot-path guard tier and a cross-framework comparative tier

## Summary

The current `benchmarks/` measures four past internal optimizations (fixes #1-4)
against monkey-patched reimplementations of code that has since moved on, and
`RESULTS.md` itself flags most numbers as "noise." Replace it with two tiers
sharing one scenario vocabulary: a **guard tier** (zero external deps, runs in CI
for visibility, never gates) that decomposes modern-di's resolve hot path, and a
**comparative tier** (an isolated nested project) that measures the same graph
against dishka, that-depends, dependency-injector, and wireup for optimization
guidance. Tooling stays on `pytest-benchmark` alone; no external service.

## Motivation

- The existing benchmarks are stale regression artifacts: they compare current
  code to hand-rolled "baseline" reimplementations of fixes #1-4, and post-#326
  (wiring-plan memoization) the baselines no longer reflect any real prior state.
- There is **no** authoritative modern-di-vs-dishka comparison anywhere — dishka
  ships no comparative suite (only a maintainer gist vs raw Python; docs make
  anecdotal `di`/`injector` claims). We want a fair, reproducible reference to
  know where modern-di's hot path stands and what to optimize next.
- Purpose (maintainer): optimization guidance + informational CI regression
  visibility + a coherent replacement for the stale suite. Explicitly **not** a
  published marketing claim.

## Design

**Axis.** All scenarios measure **sync resolution** (modern-di resolves sync;
comparing against competitors' async paths would be apples-to-oranges).
Lifecycle scenarios use **sync creator + async finalizer** — the realistic
per-request shape: `build_child_container(REQUEST)` -> resolve (sync, caches the
instance and registers the finalizer) -> `await close_async()`. Finalizers only
run for **cached** providers, so async-finalizer scenarios are cached providers.
Verified finding worth recording: modern-di is the **only** of the five that
resolves synchronously under an async finalizer — dishka, that-depends,
dependency-injector, and wireup all force an *awaited* resolve once the finalizer
is async. So C4 (the async-finalizer scenario) is an honest whole-request-lifecycle
comparison, not sync-vs-sync; C1-C3 are true synchronous resolves everywhere.

**Guard tier** — `benchmarks/`, zero external deps, decomposes the hot path so a
regression points at a specific cost:

| ID | Scenario | Isolates |
|---|---|---|
| G1 | Transient resolve, single dep, warm container | pure wiring cost |
| G2 | Cached resolve, warm cache | cache-hit lookup |
| G3 | Deep chain, depth 6, uncached | per-edge wiring |
| G4 | Wide, one object with 10 sibling deps | fan-out |
| G5 | Cross-scope resolve, REQUEST -> APP dep | `find_container` traversal |
| G6 | `build_child_container(REQUEST)` | per-request setup (scope_map build) |
| G7 | Full lifecycle: build REQUEST -> sync-init cached resolve -> `await close_async()` | real per-request cost incl. async teardown |

**Comparative tier** — the subset that maps idiomatically onto all four
competitors, each with its own setup module for the same subject-graph shape:

| ID | Scenario | Guard equiv |
|---|---|---|
| C1 | Transient resolve, single dep | G1 |
| C2 | Singleton resolve, warm | G2 |
| C3 | Deep chain, depth 6 | G3 |
| C4 | Request lifecycle: enter request scope -> sync-init resolve -> async-finalize on exit | G7 |

Each framework gets its **natural** request-scope + resource-teardown idiom (not
modern-di's scope names forced onto it); every mapping is documented in
`benchmarks/README.md` so the comparison is honest.

**Isolation (dependency handling).** The comparative tier is a **standalone,
non-packaged** nested project at `benchmarks/comparative/pyproject.toml`
(`[tool.uv] package = false`), **excluded from any uv workspace** so its
resolution never mixes competitor pins with modern-di's `dev`/`lint` toolchain.
It depends on the four competitors (pinned, for reproducibility) plus modern-di
via an editable path source (`[tool.uv.sources] modern-di = { path = "../..",
editable = true }`). Its env/lockfile are git-ignored like the root `uv.lock`.
The root `pyproject.toml` gains **only a lint-config exclude** (`benchmarks/comparative`
added to ruff `extend-exclude` and `[tool.ty.src] exclude`) so the root toolchain
does not try to type-check competitor imports it cannot resolve; **no `[project]`
dependency or dependency-group is added** — runtime identity stays genuinely
zero-dependency. The isolated project also carries its own `[tool.pytest.ini_options]`
so pytest does not inherit the root's asyncio config.

**Methodology** (baked into every benchmark): setup stays out of the timed call —
the container is built, and for warm scenarios pre-resolved once, before
`benchmark(...)`, which times only the resolve/lifecycle call; cold-resolve
scenarios use transient (uncached) providers so each timed call does the full
wiring; every benchmark asserts the resolved graph is correct so a broken no-op
setup cannot post a fake-fast number; competitor versions pinned; warm/cold
always explicit.

**Tooling & CI.** `pytest-benchmark` only (already a dev dep); no external
service. A CI job runs the guard tier on push/PR and posts a **non-gating PR
comment** with the per-scenario delta vs `main`, via
[`github-action-benchmark`](https://github.com/benchmark-action/github-action-benchmark)
with `comment-always: true` and `fail-on-alert: false`. The baseline is stored
**without a gh-pages branch**: `external-data-json-path` persisted through
`actions/cache`, written **only on `main` pushes** and read on PRs, so the delta
is a true PR-vs-main comparison. If the cache is evicted (GitHub's 7-day idle /
10 GB cap) the comment degrades to "no baseline" until main runs again —
self-healing, never fails. The **comparative tier is local-only** — never in CI,
never on a PR (four heavyweight deps, noisy, no meaningful regression-vs-main story).

**Recipes.** `just bench` -> guard tier; `just bench-compare` ->
`uv run --project benchmarks/comparative pytest benchmarks/comparative/`.

**Cleanup.** Delete `test_bench_kwargs_split.py`, `test_bench_scope_map.py`,
`test_bench_override_fastpath.py`, and `RESULTS.md`. Add a committed
`benchmarks/README.md` (living methodology: scenario definitions, per-framework
mapping, warm/cold rules, how to run). Results numbers stay **generated
artifacts**, never committed (they rot).

## Non-goals

- Published/marketing performance claims — this is internal guidance only.
- Async **resolution** benchmarks — resolution is sync-only; only finalizers are async.
- Hard CI gating / instruction-count tooling (pytest-codspeed) — rejected as
  premature external-service dependency for numbers nobody blocks on.
- Comparative tier in CI — pulling four competitors in for ungated numbers is wasteful.

## Testing

- `just bench` runs the guard tier green with no external deps.
- `just bench-compare` resolves the isolated env and runs C1-C4 across all five
  implementations; each benchmark's output-validation assert passes.
- `just check-planning` and `just lint-ci` pass (this bundle validates).
- Manual: confirm root `uv sync` is unchanged (no competitor deps pulled).

## Risk

- **Competitor pins don't resolve together** in the nested project (med likelihood,
  low impact) — isolation contains the blast radius; pin known-good versions,
  and a single framework can be dropped from the tier without touching the rest.
- **Unfair idiomatic mapping** skews comparative numbers (med × med) — mitigate by
  documenting each mapping in `benchmarks/README.md` and asserting graph
  correctness per benchmark; numbers are guidance, never published.
- **Wall-clock CI noise** makes the PR-comment delta jitter (low × med) — the
  comment is non-gating and labeled informational; a high `alert-threshold`
  avoids crying wolf, and readers treat single-digit-percent moves as noise.
