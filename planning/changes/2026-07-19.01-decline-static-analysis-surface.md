# Design: Decline the static-analysis surface, bank the strength we already hold

## Summary

Research into how DI frameworks handle **static type-safety / static analysis of
the dependency graph** (Python + cross-language) settled the question against
building anything on this surface. This change records that outcome as durable
artifacts rather than code: a `decisions/` entry rejecting a static wiring
checker, a promoted `audits/` report as the citable evidence, one non-goal line
in `docs/introduction/design-decisions.md`, and one row in
`docs/introduction/comparison.md` advertising a strength modern-di already ships
but under-sells (its injection markers type-check to `T`; FastAPI's `Depends()`
erases to `Any`). No core code changes; no `architecture/` promotion — no
capability behavior changes.

## Motivation

The competitive-borrowing backlog is heavily mined (2026-07-05 UX report ~90%
shipped; perf lane exhausted/stance). The one surface prior research explicitly
skipped was **typing / static analysis** — where cross-language frameworks
differ most. A focused, adversarially-verified research pass (108 agents, 25
claims verified 3-0/2-1, 0 refuted) established three decision-relevant facts:

1. **True static/compile-time wiring verification exists only in
   compiled-language toolchains** — Dagger's annotation processor, Google Wire's
   codegen, Koin's K2 compiler plugin (GA June 2026). Each is a heavyweight
   toolchain. Angular has one narrow compiler check; its "no provider" (NG0201)
   is runtime. .NET and Spring are runtime-startup or IDE-only. Runtime/startup
   validation — modern-di's `validate()` model — is the mainstream field
   standard, not a fallback.
2. **Koin frames its compiler plugin as a *replacement* for runtime
   `verify()`/`checkModules()`** ("you can safely remove your verify() tests").
   A static layer would **duplicate** `validate()`, not extend it.
3. **A type-checker plugin is infeasible for a conservative zero-dep library.**
   pyright refuses plugins on principle (cross-checker breakage, maintenance,
   security); `ty` — which modern-di itself uses — has no plugin system (issue
   #291 "not planned"), with Astral building framework support natively instead;
   only mypy has a plugin API, and it is "experimental… backwards-incompatible
   changes without deprecation." A plugin would serve only mypy users, be a
   permanent liability against an unstable API, not help our own `ty` toolchain,
   and duplicate `validate()`.

Meanwhile on **typed-API ergonomics** modern-di is already at the front:
`resolve(type[T]) -> T` is clean, and `integrations.py` `from_di(dep)` /
`Annotated[T, from_di(dep)]` infers to `T` — matching dishka's clean
`FromDishka[T]` and beating FastAPI's `Any`-erasing `Depends()` and
dependency-injector's base-not-subtype `Provider[Animal]`. The one cheap win the
research pointed at is already banked; comparison.md just doesn't say so.

## Design

Four documentation artifacts, no code:

1. **`planning/decisions/2026-07-19-no-static-wiring-checker.md`** (decision
   template, `status: accepted`). Decision: modern-di ships no static/compile-time
   wiring checker and no mypy/pyright/`ty` plugin; opt-in runtime `validate()`
   plus declaration-time signature parsing is the deliberate, field-standard
   model. Context = the three findings above; rationale = duplicates `validate()`
   + plugin-infeasibility + zero-dep/conservative constraints. Revisit trigger =
   `ty` ships a stable plugin API *and* a user-reported need `validate()` cannot
   meet.

2. **`planning/audits/2026-07-19-di-static-analysis-surface-report.md`** — a
   clean, cited synthesis of the research (the four axes, the verified findings
   with source URLs, the caveats incl. the Koin-vendor-source and low-star .NET
   analyzer notes). The durable citable evidence behind the decision, matching
   the shape of `audits/2026-07-18-di-concurrency-contract-report.md`.

3. **`docs/introduction/design-decisions.md`** — add one non-goal entry
   ("No static / compile-time wiring verification or type-checker plugin"),
   in the same what / why / sanctioned-alternative format as the existing
   non-goals (auto-binding, in-package integrations), pointing at `validate()`
   as the alternative and linking the decision. Only if not already present.

4. **`docs/introduction/comparison.md`** — one row/line making the typed-marker
   inference explicit: modern-di markers (`Annotated[T, from_di]`) and
   `resolve(type[T]) -> T` type-check to `T`; FastAPI `Depends()` erases to
   `Any`. A factual, cited differentiator, not a claim we can't back.

Every competitor assertion that reaches a public docs page (items 3–4) carries a
source or is phrased as our own behavior — no memory-sourced claims about other
frameworks' internals.

## Non-goals

- **No mypy/pyright/`ty` plugin, no codegen, no annotation-processor analogue** —
  the rejected surface itself.
- **No change to `validate()`, `resolve` typing, or `integrations.py`** — the
  strength is already shipped; this change only documents it.
- **No `architecture/` promotion** — no capability behavior changes; the truth
  home is untouched.

## Testing

- `just check-planning` (validates the new change/decision/audit files).
- `just lint-ci` (docs prose + planning-bundle validation).
- `just docs` builds clean; `tests/test_docs_slug_census.py` unaffected (no new
  exception classes / slugs).

## Risk

- **Inaccurate competitor claim in public docs** (low × medium): the comparison
  and non-goal touch other frameworks' behavior. Mitigation: every public claim
  is either our own behavior or carries a citation; the audit report holds the
  verified sources; time-sensitive items (Koin GA, FastAPI `Depends` typing) are
  dated.
- **Decision read as "never"** (low × low): the revisit trigger is explicit
  (`ty` stable plugin API + a real `validate()` gap).
