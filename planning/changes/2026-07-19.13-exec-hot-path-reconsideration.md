# Design: Reconsider the `exec` hot-path resolver (ethos-unbundling decision doc)

## Summary

Produce one decision-grade research report at
`planning/audits/2026-07-19-exec-hot-path-reconsideration-report.md`, plus a
matching ruling in `planning/decisions/`, that **reopens the rejected `exec`-codegen
resolver stance** — but from a single sharp angle the 2026-07-16 competitor-perf
audit did not close: that rejection was filed under *"zero-dependency library,"*
yet `exec` is stdlib and imports nothing, so dependency-purity is not actually
what is at stake. The doc unbundles "zero-dependency" from the objections that
were swept in with it and adjudicates each on its own merits. No code, no new
benchmarks: this is desk research over the evidence already on file. The report
is the sole deliverable, plus the decision ruling it produces.

## Motivation

The `exec`-codegen ceiling is the one remaining known gap in the resolve hot
path: the shipped compiled-**closure** resolver (#334) is mid-pack and the only
zero-dep pure-Python framework holding its own, trailing `dishka`/`wireup` by
1.3-1.9x on transient (C1) and deep-chain (C3) only — precisely the codegen
inline-the-dep-call frame that closures keep (`deferred.md`, the codegen-ceiling
item). That item currently calls `exec` **"a stance, not a task"** and says
"revisit only if that stance changes." This doc is the deliberate revisit of the
stance.

The reason it is worth a doc rather than a shrug: the rejection's stated ground —
*"rejected for a zero-dependency library"* (competitor-perf audit §4;
`deferred.md`) — conflates two different things. `exec` is a stdlib builtin;
`dataclasses`, `attrs`, and `cattrs` all `exec`-codegen and add no dependency.
So "zero-dependency" cannot be the real objection. The real objections —
debuggability, maintenance load, the "no magic" trust posture, free-threading
behavior, and environments that ban `exec` — are legitimate but were never
adjudicated *separately* from dependency-purity. Until they are, the stance rests
partly on a category error. This doc fixes that, without pretending the fix
changes the measured performance reality.

## Design

**Method.** Desk research, no new code or benchmarks. Sources already on file:
the two prior audits (`2026-07-16-competitor-perf-research-report.md`,
`2026-07-19-perf-readability-audit-report.md`), the nogil research report
(`2026-07-17-nogil-support-research-report.md`), the `deferred.md`
codegen-ceiling and cell-capture items, the settled `decisions/` corpus, and the
vendored rival source already in `benchmarks/comparative/.venv` (`attrs`,
`dishka`, `wireup`) for the debuggability playbook and anti-patterns.

**Structure — objection-ledger gated by an honest performance premise.** The
report is organized so the ethos reconsideration (the spine) runs against a perf
constraint it cannot wish away:

1. **Header** — Date / Method (desk research, sources above; no code/benchmarks) /
   one-paragraph Summary stating the verdict up front. Baseline commit recorded.
2. **The reframe.** State plainly that "zero-dependency" was the filed reason but
   `exec` imports nothing (stdlib precedent: `dataclasses`/`attrs`/`cattrs`).
   Name what the objection *actually reduces to* once dependency-purity is
   removed from the pile — the four rows below.
3. **The performance gate.** Restate the measured ceiling from the 2026-07-16
   audit as the binding constraint every later section is judged against: `exec`
   buys **0-4% at fixed arity, 1.3-1.9x only on construction-heavy graphs**;
   closures already capture ~80-90% of the ceiling and beat `dishka` on nothing
   but hold mid-pack. Name here the narrow forms where `exec` *does* win
   (high-arity nodes; deep-singleton chain-inlining) so the ledger knows what is
   even at stake. **Reconsidering the ethos cannot manufacture a win the
   measurement denies** — this section is the guardrail against one-sided
   advocacy.
4. **Objection ledger** — one subsection per objection. Each row follows the same
   shape: *what the 2026-07-16 audit assumed → unbundle the dependency-conflation
   → the neutralizer (or absence of one) → verdict (dissolved / real / mitigable)*.
   - **Debuggability.** The `attrs` `_linecache_and_compile` + unique-filename
     playbook (real tracebacks, PDB steps through generated code); exactly what
     modern-di would have to add to match it; `wireup`'s failure to register
     generated source in `linecache` (frame name + line number, no source line)
     as the anti-pattern to avoid.
   - **Maintainability / audit trust.** Generated-source machinery as standing
     maintenance load, and the "no magic" posture users expect from a DI layer
     they wire their whole app through. The softest and least-dissolvable row;
     the doc must not overstate the neutralizer here.
   - **Free-threading / nogil.** The modern-di-specific objection the generic
     competitor audit never covered: how `exec`-generated resolvers and their
     captured cells behave under PEP 703, against the Beta concurrency contract.
     Builds on the existing `deferred.md` cell-capture flag and the nogil
     report's out-of-scope parallel-resolution note. This is the doc's
     distinguishing technical contribution.
   - **Deployment / exec bans.** Locked-down runtimes and policies that forbid
     `exec` (e.g. RestrictedPython-style environments). Whether a pure-Python DI
     library can require `exec` on the hot path, or must keep the closure
     resolver as a fallback — which reframes `exec` as *additive* (a second
     resolver behind a fallback), not a *replacement*, and changes the
     maintenance and trust calculus of the rows above.
5. **Synthesis & recommendation.** Combine the surviving objections with the perf
   gate into a go/no-go. The recommendation is argued from the evidence; the
   ruling is the maintainer's.

**Verdict posture — genuinely open.** The doc follows the evidence to whatever
verdict it lands on, with no thumb on the scale. A pre-decided outcome would make
the reconsideration theater. Two honest landing zones: *re-decline* (if the
surviving objections still outweigh a 0-4%/1.3-1.9x win, consistent with the
measurement) or *escalate a narrow hybrid* (if one narrow `exec` form — high-arity
or deep-singleton inlining, possibly additive-behind-a-fallback — clears the bar
freshly). The doc must **land a verdict, not stay neutral**, and must not let the
dissolved dependency objection soften the perf numbers: "the ethos objection
dissolved" is never, by itself, a reason to ship.

**Decision artifact.** One ruling at
`planning/decisions/2026-07-19-<slug>.md` with the standard frontmatter
(`status` / `summary` / `supersedes: null` / `superseded_by: null`). Slug and
status follow the doc's verdict — `exec-hot-path-declined` + `status: accepted`
if it reaffirms, or a proposed-hybrid slug + `status: proposed` if the doc
surfaces a narrow form worth escalating to a spike. The ruling supersedes
nothing; the `deferred.md` codegen-ceiling item is updated to reference it (its
"a stance, not a task" line now points at an adjudicated ruling).

## Non-goals

- **No code, no benchmarks.** No prototype `exec` resolver, no new guard/comparative
  runs. If the doc concludes a hybrid warrants real modern-di numbers, that is a
  *follow-up spike*, gated by the `status: proposed` ruling — not part of this
  change.
- **Not a general codegen redesign.** The scope is the `exec` hot-path resolver
  reconsideration specifically, through the ethos-unbundling lens. Other perf
  levers (warm-singleton swap, child lazy-alloc) are settled elsewhere and are
  cited, not reopened.
- **The performance ceiling is not re-measured.** The 0-4% / 1.3-1.9x figures are
  taken as established from the 2026-07-16 audit; the doc reasons from them, it
  does not re-derive them.
- Sibling repos (modern-di-pytest, framework integrations) out of scope.

## Testing

Not a code change, so no test gate. "Landed correctly" means:

- The report exists at
  `planning/audits/2026-07-19-exec-hot-path-reconsideration-report.md` with the
  Header / reframe / perf-gate / four-row ledger / synthesis structure above.
- Every one of the four objections is adjudicated to an explicit verdict
  (dissolved / real / mitigable) — none left hanging.
- The debuggability row cites the real `attrs` mechanism and the real `wireup`
  anti-pattern from the vendored source (spot-check the file refs resolve).
- The free-threading row engages the actual `deferred.md` cell-capture flag and
  the nogil report, not a generic hand-wave.
- The synthesis lands a single go/no-go and the `decisions/` ruling matches it
  (slug + status consistent with the verdict).
- `just check-planning` passes with this change file and both artifacts present.

## Risk

- **The doc becomes advocacy for `exec`.** The unbundling insight ("it's not a
  dependency") is seductive and could tip the doc into arguing *for* `exec`.
  Mitigation: the performance gate (§3) is a first-class section every ledger row
  is judged against, and the guardrail is explicit — a dissolved objection is not
  a win. Genuinely-open posture, honest landing zones named both ways.
- **Re-litigating settled ground.** The perf-readability audit and `deferred.md`
  already touch this. Mitigation: this doc is scoped to the *one* angle they did
  not close (dependency-vs-real-objection unbundling + the four separate
  verdicts + the nogil-specific analysis); it cites the settled corpus rather
  than re-deriving it, and updates the `deferred.md` pointer rather than
  duplicating it.
- **The nogil row overreaches.** Free-threading + generated code is genuinely
  under-explored, so the row could speculate. Mitigation: anchor strictly to the
  existing cell-capture flag and the nogil report's stated Beta contract; mark
  anything beyond them as an open question for a spike, not a conclusion.
- **Verdict-shaped-by-author.** The maintainer has final say and this reconsiders
  his own stance. Mitigation: the recommendation is argued from evidence and
  labeled as a recommendation; the ruling is left explicitly to the maintainer.
