# Design: suggester owns finding a suggestion

## Summary

"Did you mean" is smeared across three modules: `suggester` holds the difflib
primitive, `ProvidersRegistry` holds the *policy* (which candidates, hierarchy
vs. typo, cap, ordering), and `exceptions` renders. This change moves the policy
into one deep `suggester.suggest(requested_type, providers)`. The registry drops
`build_suggestions` and `_hierarchy_hint` and goes back to pure storage;
`exceptions` is untouched. Behaviour and public API are unchanged; this is
Candidate 2 from the 2026-07-15 architecture review.

## Motivation

Finding a suggestion is one responsibility living in two places. The interface
that actually finds — `ProvidersRegistry.build_suggestions` plus the private
`_hierarchy_hint` — sits inside a storage module, which therefore has to know
about subclass relationships, typo distance, a `_MAX_SUGGESTIONS` cap, and the
hierarchy-before-typo ordering. `suggester` next to it is a 27-line pass-through:
`close_matches` is a one-line wrapper over `difflib.get_close_matches`, and its
7-test file (`tests/test_suggester.py`) re-asserts difflib's own documented
behaviour (cutoff, `n` cap, ordering) rather than anything this project owns.
Applying the deletion test to `build_suggestions`: deleting it does not vanish
complexity — the hierarchy/typo/cap/ordering logic has to reappear somewhere. It
belongs in the module named for it.

## Design

`suggester` gains the policy behind one interface:

```python
def suggest(requested_type: type, providers: Iterable[AbstractProvider[Any]]) -> list[Suggestion]:
    # hierarchy hints (registered subclass / base class) first, then typo matches
    # via close_matches, capped at _MAX_SUGGESTIONS, in that order.
```

`_hierarchy_hint` and `_MAX_SUGGESTIONS` move in as private members; `Suggestion`
and `close_matches` stay. `providers` is annotated **`TYPE_CHECKING`-only** and
read by duck typing on `.bound_type`/`.scope`, because a runtime import would
cycle: `suggester -> AbstractProvider -> exceptions -> suggester` (abstract
imports exceptions; exceptions imports suggester). `suggest` snapshots
`list(providers)` so it is safe against mutation-during-iteration.

`ProvidersRegistry` deletes `build_suggestions` and `_hierarchy_hint`, and with
them its `inspect` and `suggester` imports — it returns to pure storage. The two
callers pass the registry (an `Iterable[AbstractProvider]` via its `__iter__`):

```python
# container.py    suggester.suggest(dependency_type, self.providers_registry)
# factory.py      suggester.suggest(item.arg_type, registry)
```

`close_matches` stays public: it has a **second consumer** —
`UnknownFactoryKwargError` (`exceptions.py`) fuzzy-matches unknown kwarg keys
against known parameter names, unrelated to providers. That flow keeps building
its own `Suggestion` records with its own "did you mean" wording, so
`exceptions` still owns every glyph, including `_render_suggestions`.

## Non-goals

- **Rendering.** `_render_suggestions` / `_render_suggestion_lines` stay in
  `exceptions`; only *finding* moves, not formatting.
- **The kwarg-typo flow.** `UnknownFactoryKwargError` keeps constructing its own
  suggestions via `close_matches`; folding it into `suggester` would pull
  message wording out of the error that owns it.
- **Behaviour.** Byte-identical suggestions — same ordering, same cap, same
  `requested_is_class` gating. No `Suggestion` shape change.
- **A `build_suggestions` deprecation shim.** Removed outright: its return type
  (`suggester.Suggestion`) is not in `modern_di.__all__`, so it was never
  practically consumable externally; integrations use `resolve`/`add_providers`.
- **Glossary.** Adding a `Suggestion` term is Candidate 4's scope.

## Testing

- `just test-ci` — 100% line coverage, the gate.
- Delete `tests/test_suggester.py` (7 difflib re-assertions); `close_matches`
  stays covered transitively through `suggest`'s typo path and the existing
  `UnknownFactoryKwargError` tests.
- Rewrite the 2 `_hierarchy_hint` edge tests as direct `suggest()` unit tests
  (non-class bound type -> `[]`; Protocol `issubclass` `TypeError` -> `[]`),
  dropping the `_hierarchy_hint` import from `tests/test_suggestions.py`.
- Rewrite `tests/registries/test_providers_registry.py`'s `build_suggestions`
  call to `suggester.suggest(...)`.
- The 11 end-to-end suggestion tests (`tests/test_suggestions.py`, through
  `ProviderNotRegisteredError` / `ArgumentResolutionError`) stay green unchanged
  — the proof the move is behaviour-preserving.
- Promote `architecture/resolution.md` (the `build_suggestions` passage) and the
  `CLAUDE.md` `suggester.py` key-file line in the same PR.

## Risk

- **A hierarchy/typo branch drops out of coverage** after the move (low
  likelihood × low impact — the 100% gate fails loudly). Mitigated by porting
  the logic verbatim and re-pointing the two edge tests at `suggest`.
- **Import cycle** if `providers` were imported at runtime (low × high — import
  error at startup). Mitigated by the `TYPE_CHECKING`-only annotation; caught
  immediately by any import.
- **Ordering/cap regression** changing rendered messages (low × medium — visible
  in errors). Guarded by the 11 end-to-end tests that pin exact output.
